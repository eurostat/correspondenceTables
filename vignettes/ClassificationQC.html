library(correspondenceTables)

Overview

Classification quality control

The main function classificationQC() is designed to perform structural and logical quality control on hierarchical classifications. It returns a list of data frames, including an enriched version of the classification (QC_output) and additional tables flagging potential issues such as orphan codes, duplicate labels, or sequencing problems. The function takes the following arguments:

  • classification – a data frame containing the classification codes and labels.
  • lengths – a data frame defining the hierarchical structure of the codes using the columns charb and chare.
  • fullHierarchy – when TRUE, the function checks both the absence of orphan codes and the absence of childless internal nodes. When FALSE, no hierarchical completeness checks are performed.
  • labelUniqueness – when TRUE, checks that labels are unique within each hierarchical level.
  • labelHierarchy – when TRUE, checks label consistency between parents and children.
  • singleChildCode – an optional data frame defining admissible coding rules for single‑child and multiple‑child situations.
  • sequencing – an optional data frame defining admissible ordering rules for multiple‑child situations.

The function does not write output files. Export to CSV must be handled explicitly by the user outside the function.

Requirements The following auxiliary input files are expected when using the arguments lengths, singleChildCode, and sequencing.

lengths The user provides a data frame with the headers charb and chare. If these headers are missing or incorrect, the function issues a warning and automatically renames the first two columns accordingly.

singleChildCode The user provides a data frame with the headers level, singleCode, and multipleCode. If these headers are missing or incorrect, the function issues a warning and renames the first three columns accordingly.

sequencing The user provides a data frame with the headers level and multipleCode. As with singleChildCode, incorrect or missing headers are automatically detected and corrected with a warning.

In the following example, the auxiliary files are loaded to illustrate how the function handles both correctly and incorrectly specified column headers.

Auxiliary Tables for Classification Validation

The validation procedures rely on a small set of auxiliary tables that define structural constraints such as expected code lengths, single-child rules, and sequencing between levels.

We load three auxiliary tables used for classification validation. Some of the raw CSV files do not use standard column names and must therefore be normalised explicitly.

We first load the file defining the expected code lengths by level. This raw file does not provide meaningful column names, so we inspect its structure before standardisation.

Example 1: Basic quality control using hierarchy definitions

The following example applies classificationQC() to the NACE Rev.2 classification using additional parameters

In this example, the user provides: - a data frame containing the classification to be checked, and - a data frame defining the hierarchical structure of the classification through the lengths argument.

Different parameters of the classificationQC() function are used to demonstrate how structural and logical quality checks are applied to the classification.

classification <- read.csv(
  system.file("extdata", "Nace2.csv", package = "correspondenceTables")
)

lengths <- data.frame(charb = c(1,2,3,5), chare = c(1,2,4,5))

output <- classificationQC(
  classification   = classification,
  lengths          = lengths,
  fullHierarchy    = TRUE,
  labelUniqueness  = TRUE,
  labelHierarchy   = TRUE,
  singleChildCode  = NULL,
  sequencing       = NULL
)

qc_summary <- data.frame(
  Check               = c("No levels", "Orphan codes", "Childless codes"),
  Number_of_issues    = c(
    nrow(output$QC_noLevels),
    nrow(output$QC_orphan),
    nrow(output$QC_childless)
  )
)

knitr::kable(
  qc_summary,
  caption = "Summary of quality control checks"
)

Full hierarchy

When fullHierarchy = TRUE, the function checks the completeness of the hierarchical structure by applying two rules:

  • Orphan check:
    A new field in the QC output, named orphan, takes the value 1 for positions at hierarchical level j > 1 that lack a parent at the immediately higher level (j − 1), and 0 otherwise.

  • Childless check:
    A new field in the QC output, named childless, takes the value 1 for positions at hierarchical level j < k that lack a child at the immediately lower level (j + 1), and 0 otherwise.

When fullHierarchy = FALSE, no hierarchical completeness checks (neither orphan nor childless) are performed.

The labelUniqueness argument (default value: TRUE) indicates whether compliance with label uniqueness is tested.
The test outcome is reported in a new QC output column, duplicateLabel, which takes the value 1 for positions involved in duplicate labels within the same hierarchical level, and 0 otherwise.

The labelHierarchy argument (default value: TRUE) indicates whether label consistency between parents and children is tested.
The test outcome is reported in a new QC output column, singleChildMismatch, for hierarchical levels j > 1 (with no value at the highest level):

  • value 1 indicates a single child whose label does not match that of its parent;
  • value 9 indicates a child whose label matches that of its parent without being a single child;
  • value 0 indicates compliance.

Example 3: Quality control with sequencing constraints

In this final example, the sequencing parameter is used to check the ordering of multiple children within the hierarchy.

Sequencing rules are applied at hierarchical levels 3 and 4, as specified in the sequencing input table.
At these levels, the function verifies that codes corresponding to multiple children follow the expected order and flags potential gaps or inconsistencies in the sequence.


## Example: use of sequencing at levels 3 and 4

singleChildCode <- read.csv(
  system.file("extdata", "SingleChild2.csv", package = "correspondenceTables")
)

sequencing <- data.frame(
  level = c(3, 4),
  multipleCode = c("123", "123")
)

output3 <- classificationQC(
  classification   = classification, 
  lengths          = lengths,
  fullHierarchy    = TRUE,
  labelUniqueness  = TRUE,
  labelHierarchy   = TRUE,
  singleChildCode  = singleChildCode,
  sequencing       = sequencing
)
